## Root Cause

The loop is not caused by the Bevy repo itself and not by the current presence of `T04-SUMMARY.md`.

The actual cause chain is:

1. `T04` was first completed with contradictory evidence: its summary claimed progress while slice-level verification was still red.
2. `complete-slice` correctly detected that the slice was not closeable and reopened `T04`.
3. Later, `T04` was re-completed successfully on disk (`T04-SUMMARY.md` now says passed), but the `complete-slice` recovery path kept carrying forward stale context and stale warning history from the earlier failed closeout attempts.
4. Because `S06` never reached a successful `gsd_slice_complete` write, DB state remained `tasks done / slice pending`, which keeps matching the dispatch rule in `/home/fabio/.gsd/agent/extensions/gsd/auto-dispatch.js:275-286` (`summarizing → complete-slice`).
5. The closeout path then retried `complete-slice` repeatedly without real progress until stuck detection stopped auto-mode.

So the specific engine bug is:

**`complete-slice` recovery/retry is replaying stale closeout context and stale verification warnings after upstream task state changed, while dispatch continues to derive the same slice-close unit from DB state (`tasks done`, `slice pending`).**

A secondary paper-cut is that the warning surface is misleading:

- `verify-fail ... existsSync false for T04-SUMMARY.md` is stale now
- `parsers-legacy.ts require failed` is noisy fallback logging, not the blocker

## Proposed Fix

### Operational recovery now

1. Treat the current `T04-SUMMARY.md` on disk as the latest truth, not the stale `existsSync false` warning.
2. Do **not** trust the old resumed `complete-slice` session.
3. Start a fresh closeout attempt for `S06` only after confirming the latest `T04` state in DB/filesystem is aligned.
4. If auto-mode still resumes the stale closeout path, explicitly reopen/reclose the slice/task state so the dispatcher rebuilds from current artifacts instead of resumed carry-forward.

### Engine fix

Minimal code fix should target the auto-mode extension, not the game code:

1. **Invalidate resumed `complete-slice` carry-forward when any child task summary or task status changes after the prior closeout attempt.**
   - Likely touchpoints:
     - `/home/fabio/.gsd/agent/extensions/gsd/auto-prompts.js`
     - `/home/fabio/.gsd/agent/extensions/gsd/session-forensics.js`
     - `/home/fabio/.gsd/agent/extensions/gsd/auto-post-unit.js` / recovery flow

2. **Clear stale verification warnings for a unit once a later successful artifact write exists.**
   - Touchpoint:
     - `/home/fabio/.gsd/agent/extensions/gsd/auto-recovery.js`

3. **Demote or suppress the `parsers-legacy.ts` warning when `.js` fallback succeeds.**
   - Touchpoint:

4. **Optional safety guard:** before dispatching `complete-slice`, compare the timestamp/hash of inlined task summaries against the last failed closeout attempt; if changed, discard the recovery briefing and rebuild a fresh prompt.
