| D001 |  | M021 — kernel-combat / blueprint state mutation | Canonical write-path per la mutazione di state Digimon-specifico (blueprint state HashMap (UnitId, &'static str) → i32) durante una cast | **`Intent::SetBlueprintState { unit, key, value }` come unico canale di mutazione.** Le hook fn leggono `ctx.blueprint_state(unit, key) -> i32` (default 0) e scrivono esclusivamente via `ctx.enqueue(Intent::SetBlueprintState { ... })`. Nessuna mutazione diretta del HashMap dalle fn. Il payload v0 è `value: i32` semplice (non varianti tipizzate per key) — sufficiente per il roster (Predator Loop counter, Twin Core heat stack, Battery Loop charge). Validato dallo spike pass-3 (Predator Loop). Aggregator centralizzato lato `intent_applier` (D008 double-layer): la mutation finisce sia nel transition stream osservabile sia nel HashMap per la prossima read. | Read via accessor + write via Intent è il pattern già consolidato per ogni altra mutazione (damage, heal, status). Mutare il HashMap direttamente dalle hook fn romperebbe l'invariante "tutte le mutazioni passano dal transition stream" (D008), rendendo non-replicabile lo stato per replay/forensics. `i32` generico evita di forzare le hook a conoscere uno schema chiuso di key (anti-coupling kernel ↔ blueprint, P001). Spike F12 valida che read-via-predicate + write-via-Intent compongono naturalmente: il predicate `agumon::predator_active` legge il counter, l'on_kill hook scrive l'increment. Costo: due righe per hook che muta state, in cambio di full observability nel JSONL. Alternativa "key-tipizzate (enum)" rifiutata perché ricrea coupling P001 elimina; alternativa "mutazione diretta" rifiutata perché perde observability. | Sì; aggiungere varianti `Intent::IncBlueprintState`, `Intent::AddBlueprintState`, etc. è additivo se il pattern `Read+Write` per increment diventa ripetitivo. Sostituire `i32` con `enum Value { Int(i32), Bool(bool), ... }` è additivo. Promuovere a key-tipizzate è breaking. | collaborative |
| D002 | M021 post-spike pass 2/3 — superseding di D007 e D012 per riflettere il design fn-by-id validato | M021 — kernel-combat / blueprint structure | Lifecycle del `BlueprintRegistry` (Resource HashMap owner → Box&lt;dyn Blueprint&gt;) come definito in D007 — ancora load-bearing post spike pass 2/3? | **D007 superseded de facto da D032.** Lo spike pass 2/3 ha rimosso il `trait Blueprint` e il `BlueprintRegistry: Resource&lt;HashMap&lt;owner, Box&lt;dyn Blueprint&gt;&gt;&gt;` dal critical path. Sostituito da: ogni Digimon è un modulo `src/combat/blueprints/&lt;x&gt;/` con una fn `register(reg: &mut ExtRegistries)` che installa hook/predicate/selector/formula/tick/ai/cue/timeline sulle 7 registry parametriche (D031 `Registry&lt;E: ExtPoint&gt;`). Nessuna struct `Box&lt;dyn Blueprint&gt;` allocata, nessun `BlueprintRegistry` separato. `ExtRegistries` è il singolo `Resource` Bevy registrato a startup. La domanda "startup-frozen vs single-writer mutable" di D007 non si applica più — `ExtRegistries` è frozen by construction (registrazione una sola volta in `App::build`, mai mutata dopo). Hot-reload/DLC dinamico restano deferred come da D007. | D007 cristallizzava la forma "trait Blueprint + Box&lt;dyn&gt; registry". Lo spike ha mostrato che il pattern fn-by-id (D023 hook fn + D030 selector fn + D031 Registry&lt;E&gt;) è strutturalmente più semplice e copre lo stesso set di responsabilità senza trait objects, allocazioni, o registry separato. Le risposte di D007 sul lifecycle (startup-frozen, `pub(crate) register`, no hot-reload) restano corrette ma applicate al nuovo container (`ExtRegistries`). Formalizzo la supersessione qui per audit-trail invece di editare D007 in-place (rispetta append-only del DECISIONS.md). D012 (type-level enforcement blueprint↔state via `register_blueprint::&lt;B: Blueprint&gt;`) decade conseguentemente — sostituito da: ogni `register()` per-Digimon installa direttamente `app.register_blueprint_state::&lt;State&gt;()` se serve state mutabile (D034 ne è il write-path). | No, è una supersessione definitiva: ripristinare `trait Blueprint + Box&lt;dyn&gt;` post-spike richiederebbe evidence che il pattern fn-by-id ha un limite reale (nessuno emerso in 33/33 test verde sui 4 pattern stress-testati). Hot-reload futuro è additivo su `ExtRegistries` (Resource mutabile + invalidation) come era additivo su `BlueprintRegistry`. | collaborative |
| D003 | M021 S01 T02 — wiring intent_applier for DealDamage canary | architecture | System signature for `intent_applier` — exclusive `&mut World` vs Bevy query params | `intent_applier` is an exclusive system (`fn intent_applier(world: &mut World)`) that snapshots source/target unit data without Bevy query aliasing | Reading source and target unit components simultaneously via normal queries would require ParamSet or multiple separate queries, adding complexity.
