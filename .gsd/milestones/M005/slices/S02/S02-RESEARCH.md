# S02 Research — Death reaction and field exit

## Summary
S02 makes a unit reaching 0 HP visibly play its `death` frames and then fade off the field in `cargo winx`. The pure lib mapping already classifies `UnitDied -> StanceReaction::Death` (with death-precedence over `Hurt`) and exposes `StanceReaction::Death.stance_node() -> NodeId("death")`; the authored `death` node (frames 14-22) with a `death -> Exit` transition already exists in `stance.ron`. The only missing pieces are entirely in the windowed binary crate: a death event-driver that seeds the struck sprite into the `death` node (the seam currently filtered out of `drive_hurt_reactions`), and a sprite-exit mechanism (fade/despawn) that fires after the death animation completes. No lib-crate code change is strictly required — the lib half of S01 already covers death classification — though the headless test file can be extended to lock in the death/precedence contract it already asserts.

## Recommendation
1. **Death driver (windowed):** Add a dedicated death event-bridge rather than overloading `drive_hurt_reactions`. It reads `MessageReader<CombatEvent>`, collects `event.target` for events classifying as `StanceReaction::Death` (use `stance_reaction_for` directly, or `resolve_stance_reaction` per-target to inherit death-precedence), and seeds the matching sprite into `StanceReaction::Death.stance_node()` via the existing `AgumonSprite::drive_stance_reaction`. **Death must NOT be gated on `mode == Idle`** the way hurt is — death takes precedence and should interrupt any in-flight skill/hurt. `drive_stance_reaction` already unconditionally reseeds the player and graph, so calling it mid-skill is mechanically safe; just skip the `matches!(sprite.mode, Idle)` guard. Note `drive_stance_reaction` leaves `mode` unchanged (e.g. `Skill`), so a death-interrupted skill sprite keeps a stale `Skill` mode. For death that is acceptable because the sprite is about to leave the field and a new exit/fade state supersedes `mode`; but verify `sync_agumon_mode` won't re-seed the dying sprite back into the skill node on a still-active barrier (it reconciles `Skill` mode against `active_barrier` every tick in `advance_agumon_presentation`). The clean fix is to introduce a terminal marker component (e.g. `DeathExiting`) that both (a) is set by the death driver and (b) causes `advance_agumon_presentation` / `sync_agumon_mode` to skip mode-reconciliation and idle-restore for that sprite.

2. **Field exit treatment:** Use a short alpha fade-out, NOT an instant despawn, and start the fade only AFTER the `death` node completes (i.e. when `advance.exited` fires for a death-marked sprite). This preserves the documented post-KO overshoot observability (M002): the kernel deliberately lets presentation finish, so the sprite must linger through its death frames + brief fade rather than vanishing the instant `UnitDied` is read. Implement the fade as a component (e.g. `FadeOut { remaining_ticks, total_ticks }`) advanced by a small windowed system on the `PendingAnimationTicks` clock (reuse the existing fixed-rate tick the same way `advance_vfx_particles` does), lerping `Sprite.color` alpha to 0 and `commands.entity(entity).despawn()` at 0. Alpha tween on `Sprite.color` is the right mechanism (VFX particles already drive `sprite.color` via `Color::linear_rgba`, so the pattern is established in this file). Despawn-on-`death`-exit is the natural hook point: the existing `if advance.exited { ... return_to_idle }` branch in `advance_agumon_presentation` is exactly where a death-marked sprite should instead transition into fade rather than idle.

3. **Where it lives:** All new code is windowed (`src/windowed/render.rs`), respecting R002/R005 — no new lib symbols, no wgpu/winit leak. The lib mapping (`stance_reaction_for` / `resolve_stance_reaction` / `StanceReaction::Death`) is consumed as-is.

## Implementation Landscape
- `src/animation/reaction.rs` — DONE. `StanceReaction::Death`, `stance_reaction_for(UnitDied) -> Death`, `resolve_stance_reaction` death-precedence, `Death.stance_node() -> NodeId("death")`. No change needed.
- `assets/digimon/agumon/stance.ron` — DONE. `death` node frames (14,22); transition `(from: "death", to: Exit, when: TimeInNode)`. Death node exits (not loops), so `advance.exited` will fire when frames complete — this is the completion signal for starting the fade.
- `src/windowed/render.rs` — the work site.
  - `AgumonSprite` (struct, l.33): `unit_id`, `player`, `graph`, `mode` (`Idle` / `Skill`), `drive_stance_reaction` (l.143) unconditionally reseeds player+graph. Death driver reuses this. Likely add a terminal/exit marker (component or `mode` variant) — a separate `#[derive(Component)] DeathExiting`/`FadeOut` component is cleaner than extending `AgumonPlaybackMode`, since fade is a transform/color concern not an FSM playback mode, and keeps `mode_trace_fields`/`sync_agumon_mode` matches closed.
  - `drive_hurt_reactions` (l.907): the model to fork. Filters `Some(StanceReaction::Hurt)`, dedups by `event.target` into a `HashSet<UnitId>`, resolves stance graph, skips non-idle sprites. New death driver mirrors this but filters `Death`, does NOT skip non-idle, and marks the sprite as exiting.
  - `advance_agumon_presentation` (l.558): per-tick advance. The `if advance.exited` branch (l.870) currently calls `return_to_idle`. For a death-marked sprite this branch must instead start the fade (or the fade system keys off the marker independently). `sync_agumon_mode` (l.1386, called at l.627 each tick) reconciles `Skill` mode vs active barrier — must be made to skip death-marked sprites so a still-active barrier can't re-seed the dying sprite into a skill node.
  - VFX fade pattern: `advance_vfx_particles` (l.1215) shows the established alpha/despawn pattern — `age_ticks`/`ttl_ticks` countdown driven by `pending_ticks.0`, `sprite.color = Color::linear_rgba(...)`, `commands.entity(entity).despawn()` at 0. The fade system should follow this shape and run on the same `PendingAnimationTicks` clock.
  - `RenderPlugin::build` (l.290): register the new death driver (ordered like `drive_hurt_reactions`: `.after(spawn_unit_sprites).after(resolve_action_system).before(advance_agumon_presentation)`) and the fade system (ordered with/after `advance_agumon_presentation`, like `advance_vfx_particles`).
- `src/combat/resolution/apply.rs` (l.133, l.384) + types.rs (l.414): `UnitDied` is emitted with `target: resolved.target` = the dying defender. Confirmed: `CombatEvent.target` for `UnitDied` is the unit whose sprite plays death — same target semantics `drive_hurt_reactions` relies on. No combat change.
- `tests/animation/stance_reaction_mapping.rs` — already asserts `death_maps_to_death_node` and `death_takes_precedence_over_hurt_in_batch`. The headless death contract is effectively already locked; extend only if S02 adds a new pure helper (it likely won't — fade is windowed-only).

## Natural Seams
1. **Windowed death driver system** (`drive_death_reactions`): event-bridge that seeds death node + sets exit marker. Independent, mirrors `drive_hurt_reactions`. Unit-testable via small pure helpers (e.g. collecting death targets), like the existing `entered_node`/`should_release_kernel` tests in the `#[cfg(test)] mod tests`.
2. **Fade/exit component + system** (`FadeOut` + `advance_death_fade`): alpha tween + despawn on the fixed clock. Independent of the driver; the driver just marks, the fade system acts after the death node exits. Pure tween-math helper (e.g. `fade_alpha(remaining, total)`) is unit-testable headlessly within the module's test mod.
3. **Mode/exit reconciliation guard:** make `sync_agumon_mode` + the `advance.exited` idle-restore branch skip death-marked sprites. Small, but load-bearing for the interrupt-mid-skill case.
4. **(Optional) headless test extension** in `tests/animation/stance_reaction_mapping.rs` — only if a new pure helper is introduced; otherwise the death contract is already covered.

## First Proof
The highest-risk / biggest-unblocker is **death interrupting an in-flight skill without `sync_agumon_mode` clobbering it back to the skill node**. Prove this first: wire `drive_death_reactions` to seed the death node un-gated by mode, AND add the exit marker + the `sync_agumon_mode`/idle-restore skip, then confirm (via a headless unit test on the reconciliation predicate, plus a `cargo winx` death during a cast) that a unit KO'd mid-skill plays `death` and is not yanked back into the skill animation by the still-active barrier on the next tick. If this seam works, the fade is comparatively low-risk (the VFX file already proves the alpha/despawn-on-clock pattern).

## Risks / Watch-outs
- **Death-precedence vs hurt interrupt:** within a single window a unit can be both struck and killed. The lib `resolve_stance_reaction` already resolves such a batch to `Death`; but `drive_hurt_reactions` and the new death driver are two systems reading the same event stream. Ensure death wins: either run the death driver AFTER `drive_hurt_reactions` (so death reseeds over a just-applied hurt) or dedup such that a target appearing in the death set is removed from the hurt set. Ordering death-after-hurt is the simplest correct option and matches the lib's death-precedence semantics.
- **Post-KO overshoot preservation (M002):** do NOT despawn on `UnitDied` read. The kernel intentionally lets presentation complete (no liveness branching) and documents post-KO overshoot observability. The sprite must linger through its `death` frames and a short fade; start fade only after `death` node exits. Removing the sprite early would break the documented overshoot window and multi-hit loop semantics.
- **`sync_agumon_mode` re-seeding the dying sprite:** `advance_agumon_presentation` calls `sync_agumon_mode` every tick, which reconciles `Skill` mode against the still-active barrier and can re-`start_skill` the sprite — clobbering the death node. A death-exit marker that short-circuits both `sync_agumon_mode` and the `advance.exited` idle-restore branch for that sprite is required. This is the subtle failure mode; cover it explicitly.
- **`mode` staleness after `drive_stance_reaction`:** that method does not change `mode`. A death-interrupted skill sprite keeps `mode = Skill`. The exit marker must take precedence over `mode` in all per-tick branches, or convert/clear mode on death.
- **Dep-gating (R002/R005/R016):** keep all death-fade code in the windowed crate. The fade uses `Sprite.color`/`Transform`/`commands.despawn` — all windowed. No new lib symbol; the lib mapping is consumed read-only.
- **R010 (UI from events):** the death driver reads `CombatEvent` and mutates only presentation components/entities — it must never touch `CombatState`. Same discipline as `drive_hurt_reactions`.
- **Determinism (R004):** the fade is a windowed visual on a wall-clock-derived fixed tick; it must not feed back into the kernel/barrier or alter event-observable state. Keep it strictly downstream of presentation.
