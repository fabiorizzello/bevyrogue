# S03 RESEARCH — Hit feedback: flash, shake, and canvas damage numbers

## Summary
S03 adds three transient, on-hit presentation effects to the windowed view: a **color flash** and a **positional shake** of the struck `AgumonSprite`, plus **floating damage numbers rendered in world-space on the pixel canvas** over the target. Combat already emits `CombatEventKind::OnHitTaken { amount }` (defender-scoped, fires on every non-revive hit) — this is the damage value source. All three effects are driven from `MessageReader<CombatEvent>` (R010) following the exact `drive_hurt_reactions` event-bridge pattern S01 established in `src/windowed/render.rs`; none mutate `CombatState`. The biggest unknown is the canvas damage number: damage numbers exist today only as an **egui side-panel** painter (`render_floating` in `src/ui/combat_panel/widgets.rs`, fixed panel coordinates), and there is **zero `Text2d`/world-space text usage anywhere in `src/`** — that rendering is greenfield.

## Recommendation

### 1. Damage flash (lowest risk — ride existing infra)
Tint `Sprite.color` of the struck `AgumonSprite` for a brief window, then restore to `Color::WHITE`. A windowed-only `FlashState { remaining_ticks }` field added to `AgumonSprite` (or a sibling `Flash` component on the same entity) set from a new event-bridge system, then decayed and applied in `advance_agumon_presentation`. **Critical:** `advance_agumon_presentation` already owns `&mut Sprite` and rewrites `texture_atlas.index` each tick but does **not** currently write `Sprite.color` for the AgumonSprite (only VFX particles get color writes, in `advance_vfx_particles`). So the flash is the sole writer of the AgumonSprite's color — apply the tint at the end of the per-tick block, restoring to white when `remaining == 0`. Use the existing `PendingAnimationTicks` budget for decay so flash duration tracks the same fixed-rate clock as everything else (precedent: `TargetHurtState`/`HURT_FRAMES` already does a 12-frame countdown off `OnHitTaken`).

### 2. Shake (low risk — but must NOT fight the rest position)
Offset `Transform.translation` with a decaying jitter, then return to the **layout rest position**. The rest X/Y is computed in `spawn_unit_sprites`: `x = if enemy { 200.0 } else { -200.0 }`, `y = 0.0`, z=0, scale `SPRITE_DISPLAY_SCALE (0.4)`. **Watch-out:** `advance_agumon_presentation` reads `transform.translation` to anchor VFX (mouth/caster anchors), so the shake offset will perturb VFX anchoring while active — acceptable for a brief shake, but the offset MUST be fully removed when the shake ends. Recommended: store the *rest* position in the shake state at spawn (or recompute from `Team`), apply `rest + jitter` each tick, and on completion hard-set translation back to rest (do not accumulate). Apply in `advance_agumon_presentation` (it already holds `&mut Transform` via `sprites.p0()` query `(&mut AgumonSprite, &mut Sprite, &Transform)` — note `Transform` is currently read-only there; it must become `&mut Transform`).

### 3. Floating damage numbers on the canvas (highest risk — greenfield world text)
Spawn a Bevy **`Text2d`** entity at the target sprite's world position on each `OnHitTaken`, carrying the `amount`, then float it up and fade it (`TextColor` alpha) and despawn. Recommended approach: a new windowed-gated component (e.g. `CanvasDamageNumber { remaining_ticks, ... }`) on a `Text2d` entity, spawned by a new event-bridge system that resolves the target's world XY the same way existing VFX code does (`find_sprite_xy(&sprites, target)` already exists in `render.rs`), placed at a small +Y offset above the sprite at a z above particles (`VFX_PARTICLE_Z = 1.0`, so use z ≥ 2.0). A per-tick advance system floats it up and fades alpha, despawning at end-of-life. Do NOT reuse the egui `FloatingDamage` component for this directly (it is consumed by the egui panel render); but the `OnHitTaken { amount }` event already carries everything, and you may mirror its `DamageKind`-based prefix/coloring (`Normal`/`Weak`/`Resist`/`Break`) if you also key off `OnDamageDealt { amount, kind, .. }` for the kind. Simplest honest scope: white number from `OnHitTaken.amount`; optionally enrich color from the co-emitted `OnDamageDealt.kind`.

## Implementation Landscape

### What exists
- **Event-bridge pattern** — `drive_hurt_reactions` in `src/windowed/render.rs:907` reads `MessageReader<CombatEvent>`, filters via pure `stance_reaction_for`, dedups targets into a `HashSet<UnitId>`, and drives the struck sprite. Registered in `RenderPlugin::build` (`render.rs:310-317`) ordered `.after(spawn_unit_sprites).after(resolve_action_system).before(advance_agumon_presentation)`. Each consumer gets its own `MessageReader` cursor (MEM065) — multiple systems can read the same bus without starving.
- **Sprite mapping** — `AgumonSprite { unit_id, .. }` (`render.rs:34`); resolve `UnitId → entity` by iterating the query and matching `unit_id` (see `find_sprite_xy`, `render.rs:1135`). One sprite per unit, spawned in `spawn_unit_sprites` (`render.rs:514`).
- **Per-tick advance** — `advance_agumon_presentation` (`render.rs:558`) owns `(&mut AgumonSprite, &mut Sprite, &Transform)` and runs N ticks per frame from `PendingAnimationTicks` (fixed-rate `AnimationClock`, MEM059/MEM064). It writes `texture_atlas.index` but NOT `Sprite.color` for AgumonSprite, and reads (not writes) `Transform`.
- **Camera/canvas** — `setup_camera` (`render.rs:330`): single `Camera2d` + `Hdr` + `Bloom::NATURAL` + `Tonemapping::TonyMcMapface` + `DebandDither`. World-space units; sprites at x=±200, scale 0.4 → ~205px. `Text2d` will render in this same world space (MEM082: HDR is the `Hdr` component, not `Camera{hdr:true}`).
- **Damage value source** — `CombatEventKind::OnHitTaken { amount }` (`src/combat/observability/events.rs:92-95`), fired for the defender on every non-revive hit; companion `OnDamageDealt { amount, kind, damage_tag, .. }` (`events.rs:38`) carries `DamageKind`. `stance_reaction_for` (`src/animation/reaction.rs:42`) maps `OnHitTaken → Hurt`.
- **Existing damage-number rendering (egui only)** — `FloatingDamage` ECS component (`src/combat/observability/floating.rs:9`: `target/amount/kind/spawn_time`), spawned in combat pipeline (e.g. `single_target.rs:256`), decayed by `decay_floating_damage` (`floating.rs:17`, registered in `src/main.rs:58`). Rendered ONLY in the egui side panel by `render_floating` (`src/ui/combat_panel/widgets.rs:511`) at fixed panel coordinates (`x = panel_w * 5/6 + jitter`), NOT world-space. `compute_floating_damage_view` (`src/ui/combat_panel/mod.rs:418`) projects it for display. `FLOATING_LIFETIME_SECS = 1.2`, `FLOATING_MAX_LIVE = 8`.
- **Hurt blink precedent** — `TargetHurtState` + `observe_target_hurt`/`tick_target_hurt_state` (`combat_panel/mod.rs:311-415`) is a tick-countdown projection driven by `OnHitTaken`, headless-unit-tested in `tests/windowed_only/windowed_target_hurt.rs`. This is the exact shape a flash/shake decay timer should follow and is the model for the headless-testable seam.

### What is missing
- No `Text2d` / world-space text anywhere in `src/` — canvas number rendering is new.
- No `Sprite.color` flash on AgumonSprite, no `Transform` jitter on AgumonSprite.
- No event-bridge systems for flash / shake / canvas-number (must be added + registered in `RenderPlugin`).

## Natural Seams
Three independent units plus one shared piece:
1. **Flash** — `Sprite.color` tint + decay on the struck AgumonSprite. Self-contained; only touches color (currently unwritten for AgumonSprite). Smallest.
2. **Shake** — `Transform.translation` jitter + decay-to-rest. Independent of flash; must change `advance_agumon_presentation`'s `&Transform` to `&mut Transform`. Needs the rest position.
3. **Canvas damage numbers** — spawn `Text2d` at target world XY, float-up + fade + despawn. Fully independent entity lifecycle (like VFX particles), highest novelty.
- **Shared:** a windowed-only "struck this window" signal derived from `OnHitTaken` (event-bridge reader). Flash and shake both want the struck `UnitId` set; they can share a single bridge system that arms both a `FlashState` and `ShakeState` (or each reads the bus independently per MEM065). The canvas number bridge additionally needs `amount` (and optionally `kind` from `OnDamageDealt`), so it is naturally its own reader.

## First Proof
**Canvas world-space damage number rendering.** It is the only effect with no prior art in `src/` (no `Text2d`), it is the literal headline of the slice ("floating damage number on the canvas over the target"), and it is the biggest unblocker: proving a `Text2d` spawns at the target's resolved world XY, floats, fades, and despawns under the HDR/bloom camera de-risks the entire slice. Flash and shake are mechanically trivial once the event-bridge + per-tick-decay scaffold (already proven by `TargetHurtState`) is in place. Suggested proof: an `OnHitTaken` writes a `CanvasDamageNumber` `Text2d` entity at the target sprite position; assert (headless, see Verification) that a number entity with the correct text/amount and a finite life is spawned and decays/despawns.

## Risks / Watch-outs
- **Fighting the stance-driven Sprite/Transform.** `advance_agumon_presentation` rewrites `texture_atlas.index` every tick. It does NOT write AgumonSprite `Sprite.color` today, so flash is the sole color writer — but confirm no other system tints AgumonSprite. For shake, `Transform` is currently read-only in that system; making it `&mut` and applying a non-zero offset will move the VFX mouth/caster anchors (`advance_agumon_presentation` and `advance_vfx_particles` read live `transform.translation`). Keep the shake brief and ALWAYS restore to the exact rest position (store rest, set `rest + jitter`, never accumulate) so it does not permanently offset the sprite or corrupt anchoring.
- **Rest position sourcing.** Rest is `x = ±200` by `Team`, `y = 0` (`spawn_unit_sprites`). If multi-slot positioning lands later this becomes stale — capture rest at spawn time into the shake state rather than hardcoding.
- **Effects must not interrupt the stance/skill FSM.** S01 already guards: hurt reaction only fires on an idle sprite. Flash/shake/number are pure overlays (color/offset/separate entity) and should fire regardless of cast state since they don't touch `AnimGraphPlayer` — but keep them orthogonal to `mode`/`player` so they never perturb the kernel-barrier release logic.
- **Damage value sourcing.** `OnHitTaken { amount }` is the canonical per-hit defender value and already drives the hurt blink; use it. If kind-colored numbers are wanted, the `kind` lives on the co-emitted `OnDamageDealt { amount, kind }` (attacker-companion) — correlating the two within a window adds complexity; simplest honest scope is a white number from `OnHitTaken.amount`.
- **Dedup/multi-hit.** Multiple `OnHitTaken` in one window (multi-target/bounce) should each produce their own number (per-hit), but flash/shake should dedup per target (like `drive_hurt_reactions`' `HashSet<UnitId>`) so a double-hit doesn't stack offsets. Decide per effect.
- **Dep-gating (R002/R005/R016).** All new code is windowed-gated. The egui `FloatingDamage`/`decay_floating_damage` path already lives behind the combat module but `render_floating` is `#[cfg(feature = "windowed")]`; keep the new canvas-number path entirely in `src/windowed/render.rs` (or a windowed-gated module) so no Bevy 2d/text types leak into the headless lib. Do not let the new world-text path regress the existing egui panel numbers (both can coexist).
- **Lifetime/timing model.** The egui path uses wall-clock `time.elapsed_secs()` + `FLOATING_LIFETIME_SECS`; the windowed sprite/VFX path uses the fixed-rate `PendingAnimationTicks` budget. Pick one consistently per effect — recommend the tick budget for flash/shake (matches sprite cadence) and either model for the number (tick budget keeps it in lockstep with VFX).
